# Internal FastAPI Guidelines

## 1. 项目定位
- 默认目标是生成**完整、可运行、可维护**的网站项目，而不是一次性演示代码。
- 当需求只需要静态展示时，优先生成多文件前端项目（如 `index.html`、`styles.css`、`script.js`）。
- 当需求涉及服务端逻辑、数据处理、接口、认证、持久化或 Python 集成时，使用 FastAPI。

## 2. FastAPI 项目结构
- Python 项目使用 `uv` 和 `pyproject.toml`。
- 推荐结构：
  - `app/main.py` 作为应用入口
  - `app/routers/` 放路由
  - `app/services/` 放业务逻辑
  - `README.md` 提供运行说明
- 路由层负责请求和响应组织；业务逻辑尽量放在 `services` 中，避免全部堆在 `main.py`。

## 3. 前后端协作原则
- 先判断是否真的需要后端，不要为简单展示页强加 API。
- 如果使用 FastAPI，接口命名应清晰稳定，例如 `/health`、`/api/trails`、`/api/search`。
- 前端调用接口时，应展示加载状态、错误提示和空结果提示。
- 页面上对用户可见的文本优先使用中文，除非用户明确要求其他语言。

## 4. 页面设计规范
- 页面应具备清晰的层次：页头、主内容区、辅助信息区、页脚。
- 优先使用语义化 HTML：`header`、`main`、`section`、`article`、`footer`。
- 列表型内容优先使用卡片布局，支持桌面与移动端显示。
- 颜色与排版需要统一，不要使用杂乱无章的默认样式。
- 页面要有明确主按钮、辅助按钮、状态提示和空状态展示。

## 5. 交互型网站与数据展示页面建议
- 当页面包含搜索、筛选、排序、分页、表单提交等交互时，前端应提供明确反馈。
- 推荐覆盖以下状态：加载中、请求失败、无结果、结果为空、提交成功。
- 列表类页面应支持清晰的信息分组，避免内容堆叠。
- 如果页面依赖外部数据源，应优先选择可直接运行、依赖简单的实现方案。
- 对需要可视化的场景，可根据任务选择表格、卡片、时间线、图表或地图，但不要预设只有某一种展示方式。

## 6. 代码质量要求
- 所有 Python 代码遵守 PEP 8。
- 禁止使用 `print()`，统一使用 `logging`。
- 尽量选择最小可行实现，但不能牺牲可运行性与项目结构清晰度。
- 生成结果应尽量包含可运行说明，确保用户拿到项目后能快速启动。

## 7. 生成结果要求
- 默认输出应是**项目**，不是孤立代码片段。
- 静态网站通常至少包含：
  - `index.html`
  - `styles.css`
  - `script.js`
  - `README.md`
- FastAPI 网站通常至少包含：
  - `pyproject.toml`
  - `app/main.py`
  - `app/routers/`
  - `app/services/`
  - `README.md`

## 8. Agent 行为建议
- 当用户要求构建网站项目时，应优先结合本地规范与外部资料，选择合适架构。
- 如果本地文档给出了明确项目结构或页面规范，应优先遵循。
- 如果已有可复用 Skill，可参考其模式，但需要根据当前任务做调整，而不是机械套用。
